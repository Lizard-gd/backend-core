## 🆚 Сравнение: new внутри сервиса vs DI через конструктор

### ❌ *BAD:* new InMemoryLeadRepository() внутри класса

🚨 Проблемы:
1. 🧪 Невозможно подставить mock в тестах
2. 🔄 Невозможно заменить реализацию без изменения кода
3. 🕵️ Скрытая зависимость — не видно, что нужно для работы

### ✅ *GOOD:* DI через конструктор
🐛 **Пример:**
```
public class LeadService {
    private final LeadRepository repository;

    public LeadService(LeadRepository repository) {
        this.repository = repository;
    }
}
```

🌟 Преимущества:
1. 🎭 В тестах передаём mock(LeadRepository.class)
2. 🏭 В production передаём InMemoryLeadRepository
3. 🗄️ Также передаём JpaLeadRepository
4. 👁️ Зависимость явная — видно в конструкторе

### 🔄 Что такое IoC простыми словами

 * **Inversion of Control (инверсия управления)** — это когда класс НЕ создаёт свои зависимости сам, а получает их извне. Контроль переворачивается: не сервис решает, какой репозиторий использовать, а тот, 

#### "Не звоните нам, мы позвоним вам" ☎️

* Традиционно: LeadService → создаёт → InMemoryLeadRepository
(класс сам решает, что использовать)

* С IoC: Main/Test → создаёт → Repository → передаёт в → LeadService
(контроль переворачивается — внешний код решает)

🏠 **Аналогия:**
* Вы не строите розетку в стене сами (BAD). Электрик (DI) приходит и подключает то, что нужно (GOOD). Вы просто говорите: "Мне нужна розетка" — а какую именно, решает тот, кто её подключает.
